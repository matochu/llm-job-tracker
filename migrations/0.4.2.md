## Step 1 — Seed Browser MCP patterns

**Target file:** `config/browser-patterns.md`.

**Check:** this step is satisfied when `config/browser-patterns.md` exists.
- If found -> skip.
- If not found -> apply.

**Apply:**

1. In plugin mode, copy `$CLAUDE_PLUGIN_ROOT/config/browser-patterns.md` to `config/browser-patterns.md` if that source exists.
2. In workspace mode, copy the managed workspace source `config/browser-patterns.md` if present after CLI update.
3. If no source is available, create the file with the following content:

```md
# Browser MCP Patterns

Use Browser MCP only for sources that need session state, JavaScript rendering, filters, or login.

## Stable Calls

- For Playwright/Chrome `evaluate`, pass a JavaScript `function`, not a raw expression.
- Do not assume accessibility snapshot refs are valid selectors for a different MCP call. If a ref fails, switch to locator-based selection.
- Prefer stable locators: role/name, text, URL fragments, form labels, and fuzzy class selectors such as `[class*="jobs"]`.
- For cookie overlays and modals, close them with locator-based button clicks such as text/role matches for `Accept`, `Reject`, `Close`, or localized equivalents.
- If login is required, open the page in Browser MCP and wait for the user to authenticate manually.

## Avoid

- Do not replace LinkedIn, Djinni, or browser-required portfolio-board checks with plain web-search snippets.
- Do not rely on brittle full class names from LinkedIn, Djinni, or generated ATS pages.
- Do not click submit/apply/send/connect controls unless the active skill explicitly allows it and the user has confirmed the action in the same run.
```

**Success condition:** `config/browser-patterns.md` exists and contains `# Browser MCP Patterns`.

**Failure condition:** the file is still missing or does not contain the heading after the write. Stop and report. Do not bump `.migrated-version`.
